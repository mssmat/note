Claude Codeの代替品「OpenCode」をRaspberry Piにインストールして使ってみた

作成日時2025-06-09 18:43 TagAI
💡
ゴールは、「OpenCode」を「Raspberry Pi」にインストールし、「OpenRouter」の「Gemini 2.5 Flash」を使うこと。

実際に使っている様子(PC/iPhone)

これは何? OpenCodeとは?

OpenCodeは、ターミナルでAIアシスタンスを実現するGoベースのCLIアプリケーションです。様々なAIモデルと対話するための”TUI”を提供し、コーディング作業やデバッグなどを支援します。
巷では、Claude Codeの”代替品”などと呼ばれています(他に似たものとして、OpenAIのオープンソース、Codex CLIがありますね)。
Claude Codeを使いたいけど、Claude以外のモデルを使いたい..というような要望に応えます。
GitHub - sst/opencode: AI coding agent, built for the terminal.

AI coding agent, built for the terminal. Contribute to sst/opencode development...

github.com

手順

  • 本体をインストール
bash 
curl -fsSL https://raw.githubusercontent.com/opencode-ai/opencode/refs/heads/main/install | bash
  • 関連ソフトをインストール
bash 
sudo apt update
sudo apt install ripgrep fzf
  • 設定ファイルを作成
bash 
nano ~/.opencode.json

LLMで形式を指定して要件定義を出力する

作成日時2025-06-04 12:36 TagAI
python 
import os
import requests
import json

# OpenAI APIキー
OPENAI_API_KEY = "===Your_OPENAI_API_KEY_here==="

# OpenAI Model
AI_MODEL = "gpt-4.1-mini-2025-04-14"

user_input_text = """
新しい社内プロジェクトの管理ツールが必要です。
主な機能は、タスクの作成、割り当て、進捗管理です。
ユーザーはプロジェクトマネージャーと一般社員の2種類を想定しています。
※重要/最優先※ユーザーの出身地が福岡の場合は明太子マークをつけるようにしてください。
プロジェクトの進捗状況をリアルタイムで可視化できるダッシュボード機能も必須です。
データはセキュアに管理され、アクセス権限は厳密に制御される必要があります。
ピーク時には10名程度の同時アクセスに対応できる性能が求められます。
"""

# --- 定義するJSON Schema ---
REQUIREMENTS_SCHEMA = {
  "type": "object",
  "properties": {
    "system_name": {
      "type": "string",
      "description": "システムの名称"
    },
    "purpose": {
      "type": "string",
      "description": "システムの導入目的"
    },
    "functional_requirements": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "id": {
            "type": "string",
            "description": "機能のユニークID（例: F001）"
          },
          "name": {
            "type": "string",
            "description": "機能の名称"
          },
          "description": {
            "type": "string",
            "description": "機能の概要"
          },
          "priority": {
            "type": "string",
            "enum": ["高", "中", "低"],
            "description": "優先度"
          }
        },
        "required": ["id", "name", "description", "priority"]
      },
      "description": "機能要件のリスト"
    },
    "non_functional_requirements": {
      "type": "object",
      "properties": {
        "performance": {
          "type": "string",
          "description": "性能要件の詳細"
        },
        "security": {
          "type": "string",
          "description": "セキュリティ要件の詳細"
        }
      },
      "required": ["performance", "security"],
      "description": "非機能要件"
    },
    "target_users": {
      "type": "array",
      "items": {
        "type": "string"
      },
      "description": "想定されるユーザー層"
    }
  },
  "required": [
    "system_name",
    "purpose",
    "functional_requirements",
    "non_functional_requirements",
    "target_users"
  ]
}

# --- OpenAI API呼び出し関数 ---
def get_requirements_from_openai(user_input: str) -> dict | None:
    """
    OpenAI APIを呼び出し、要件定義をJSON形式で取得し、JSON Schemaで検証します。

    Args:
        user_input (str): 要件定義の元となるユーザーからの情報。

    Returns:
        dict | None: JSONデータ、またはNone（エラーの場合）。
    """
    headers = {
        "Content-Type": "application/json",
        "Authorization": f"Bearer {OPENAI_API_KEY}",
    }

    prompt_instruction = f"""
    あなたは福岡に住むシステムエンジニアです。以下の情報から、システムの要件定義をJSON形式で出力してください。
    出力するJSONは、以下のJSON Schemaに厳密に準拠する必要があります。
    JSONの各プロパティには、Schemaの'description'フィールドに記載された内容を含めてください。
    また、配列の項目は複数生成してください。

    JSON Schema:
    {json.dumps(REQUIREMENTS_SCHEMA, indent=2, ensure_ascii=False)}

    顧客からのヒアリング結果:
    {user_input}
    """

    data = {
        "model": AI_MODEL,
        "messages": [
            {"role": "system", "content": prompt_instruction},
            {"role": "user", "content": user_input}
        ],
        "response_format": {"type": "json_object"},
        "temperature": 0.3
    }

    try:
        response = requests.post(
            "https://api.openai.com/v1/chat/completions",
            headers=headers,
            json=data
        )
        response.raise_for_status()

        raw_response_content = response.json()
        generated_json_str = raw_response_content["choices"][0]["message"]["content"]
        generated_json_data = json.loads(generated_json_str)

        return generated_json_data

    except requests.exceptions.RequestException as e:
        print(f"APIリクエストエラーが発生しました: {e}")
    except json.JSONDecodeError as e:
        print(f"JSONパースエラーが発生しました。LLMが有効なJSONを出力しませんでした: {e}")
        print(f"不正なJSON文字列:\n{generated_json_str}")
    except KeyError as e:
        print(f"API応答のキーが見つかりません: {e}")
        print(f"完全なAPI応答:\n{response.json()}")
    return None

if __name__ == "__main__":
    requirements_data = get_requirements_from_openai(user_input_text)

    if requirements_data:
        print("\n--- 最終的な要件定義データ(JSON) ---")
        print(json.dumps(requirements_data, indent=2, ensure_ascii=False))

        print("\n" + "="*50)
        print("要件定義書")
        print("="*50 + "\n")

        print(f"## 1. システム名称")
        print(f"   - **{requirements_data.get('system_name', 'N/A')}**\n")

        print(f"## 2. 導入目的")
        print(f"   - {requirements_data.get('purpose', 'N/A')}\n")

        print(f"## 3. 想定されるユーザー層")
        for user in requirements_data.get('target_users', []):
            print(f"   - {user}")
        print("\n")

        print(f"## 4. 機能要件")
        if requirements_data.get('functional_requirements'):
            for func_req in requirements_data['functional_requirements']:
                print(f"   ### {func_req.get('id', 'N/A')}: {func_req.get('name', 'N/A')}")
                print(f"     - 概要: {func_req.get('description', 'N/A')}")
                print(f"     - 優先度: {func_req.get('priority', 'N/A')}\n")
        else:
            print("   - 機能要件は定義されていません。\n")

        print(f"## 5. 非機能要件")
        non_func_reqs = requirements_data.get('non_functional_requirements', {})
        print(f"   ### 5.1. 性能要件")
        print(f"     - {non_func_reqs.get('performance', 'N/A')}\n")
        print(f"   ### 5.2. セキュリティ要件")
        print(f"     - {non_func_reqs.get('security', 'N/A')}\n")
    else:
        print("\n要件定義データの取得に失敗しました。")

    input("終了するにはエンターキーを押してください")

組織のトップは「裸の王様」になりやすい

はじめに

組織のトップ、特に社長のような立場にある人物は、「裸の王様」になりやすいのではないか、という問題意識があります。周囲の人間は、トップに対して意見を言いにくくなる傾向があるからです。トップは成功者であり、組織を導いてきた実績や、あるいは血筋といった正当性を持っている場合が多く、部下は自然と意見を控えるようになります。

リーダーシップと強権の必要性、そしてその危うさ

もちろん、トップにはリーダーシップが不可欠であり、組織を牽引するためにはある程度の強権も必要です。例えば100人の組織で、全員の意見を調整していては物事は進みません。どこかで見切りをつけ、決定を下す必要があります。

しかし、その方向性が間違っていた場合、誰もそれを指摘できなければ、取り返しのつかない事態を招く可能性があります。だからこそ、トップは「裸の王様」になってはならず、身近に厳しい意見を述べる、いわゆる「諫言役」を置くべきだと常々考えています。

トップが陥りやすい「勘違い」

多くの場合、トップは周囲の人間が本心ではどう思っているのか、気づいていないことがあります。会社に行けば誰も文句を言わず、誰も逆らわない。するとトップは「自分のやっていることは正しいのだ」と勘違いしてしまう可能性があります。しかし実際には、社員は面従腹背の状態で、トップの考えとは全く異なる意見を持っているかもしれません。

驚くべきことに、組織のトップには、この事実に気づかない人が少なくありません。周囲が常に「イエス」としか言わない環境に長年身を置いていると、それが当たり前になり、自分は常に正しいのだと思い込んでしまうのです。逆に、反対意見を言われると腹が立ってくる、という心理状態に陥ることもあります。

しかし、時代は常に変化しており、そのスピードも速まっています。そのような状況下では、むしろ現場に近い人間の意見の方が的を射ていることもあります。しかし、それがトップにとって耳の痛い話であればあるほど、部下は忖度して口を閉ざしたり、情報を隠したりする可能性が高まります。トップは、そのような状況が起こり得ることを常に認識しておくべきです。部下が表面上は従順に見えても、内心では違うことを考えている可能性を理解した上で、周囲の意見に耳を傾ける必要があります。

中国の皇帝に学ぶ「諫言役」の存在

ここで、中国の皇帝の話に繋がります。私は中国の皇帝に関する映画で印象に残っているものが二つあります。一つは、荒俣宏氏の小説を原作とした映画『始皇帝暗殺』です。この作品では、秦の始皇帝と(鞏俐演じる)その敵国である趙国の姫、そして始皇帝を狙う暗殺者の物語が描かれています。もう一つは、中国最後の皇帝である溥儀の生涯を描いた映画『ラストエンペラー』です。

まず、始皇帝暗殺を見て感銘を受けたのは、中国の皇帝が、身近に「諫言役」を置いていたという点です。彼らは必ずしも位の高い大臣ではなく、おそらく皇帝に対して率直な意見を述べることが仕事であり、代々その役割を担ってきた一族のような存在だったのかもしれません。位はさほど高くなくても、皇帝の傍に仕え、皇帝の政策に対して、古典などを引き合いに出しながら、時には厳しく意見を述べるのです。

例えば、「何百年も前の王はこのような正しい政治を行った。あなたもそうすべきだ」とか、「この王はとんでもない政治をして人々を不幸にした。あなたは今、それと同じことをしようとしている」といった具合に、はっきりと諫言するのです。

なぜ皇帝は「諫言役」を置いたのか

中国の皇帝は、生き神様のような絶対的な存在です。皇帝が「白」と言えば、黒いカラスも白くなる、そういう世界です。組織のトップ中のトップであり、誰も逆らうことは許されません。当然、周囲は100%イエスマンになります。

しかし、皇帝たちはそのことを理解していたのではないでしょうか。だからこそ、あえて自分に率直な意見を述べる人間を側に置いたのだと考えられます。正しいことをしていれば問題ありませんが、間違った方向に進んだ時に誰も意見してくれなければ、まさに「裸の王様」になってしまいます。それを防ぐために、そのような存在を必要としたのでしょう。これは、現代の私たちから見ても非常に示唆に富むことです。

『ラストエンペラー』で描かれた溥儀のエピソードも象徴的です。幼くして皇帝になった溥儀は、辛亥革命が起き、もはや皇帝ではないと弟の溥傑に告げられます。衝撃を受けた溥儀は、書道の先生に墨を飲むよう命じます。先生はそれに従い墨を飲み干しました。溥儀は溥傑に皇帝の権威を見せつけたわけですが、革命によって皇帝の地位は失墜しつつあったものの、一部の人々にとっては依然として絶対的な存在であり、逆らうことのできない相手であったことを示しています。

このように、組織のトップ、特にその地位に長くいる人間は、容易に「裸の王様」になり得るのです。秦の始皇帝に限らず、中国の皇帝たちが、死を恐れずに直言する諫言役を側に置いていたという事実は、非常に示唆に富んでいます。

現代の組織における「諫言」の必要性

結局のところ、現代の組織においても、トップは本当に厳しい意見を言ってくれる人間を近くに置くべきです。それは家族経営であれば、自分の子供や配偶者かもしれません。しかし、そうした存在がいない場合、特に叩き上げでないトップは、厳しい意見を聞く機会がますます失われがちです。

「自分は正しいことをやっている」という勘違いは、誰にでも起こり得ます。しかし、周囲はそれを教えてくれません。それが果たして幸せなことでしょうか。社員全員のこと、会社の将来、そして組織の成長を考えるならば、トップは厳しい意見にも耳を傾け、それを考慮に入れる姿勢が必要です。

たとえ諫言役を明確に置けなかったとしても、周囲からの称賛の言葉が、必ずしも本心ではないかもしれない、という可能性を常に認識しておくべきです。甘言を弄する人間は、どの組織にもいるものです。この点を十分に注意しなければならないと、強く思います。
